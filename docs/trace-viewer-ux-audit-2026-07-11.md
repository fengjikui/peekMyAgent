# Trace Viewer 信息架构与多 Agent 展示审计

更新时间：2026-07-11
审计基线：`a11aaeb31259bd0acc6b818f0260564ade1f46e4`

## 第一阶段落地结果

审计后首先修复了“视觉折叠但 DOM 全量生成”的主性能问题，并在同一条 1,536 请求、106 个子 Agent Trace 上复测：

| 状态 | 审计基线 | 第一阶段结果 |
| --- | ---: | ---: |
| 完整 Trace 载入、看板折叠 | 73,243 DOM 节点 / 16,377 按钮 | 501 DOM 节点 / 41 按钮 |
| 106 子 Agent 看板展开 | 3,024 按钮，全部分支与事件已生成 | 1,176 DOM 节点 / 170 按钮；首批 24 分支 / 80 事件 |
| 单分支展开 | 所有分支步骤预先生成 | 仅为当前分支新增步骤 DOM |

同时明确区分“子 Agent 实例数”和“子 Agent 请求数”，并把 Skill 提示词注入识别为 Harness 内部请求，避免产生虚假的用户 Turn。以上是当前已实现行为。

第二阶段继续落地了审计中的导航建议：

- 中栏按自身宽度响应，1280px 三栏状态下 541px 的主栏仍保持标题单行；折叠左栏后主栏与 Raw 按原内容比例共同扩展。
- 多 Agent 看板增加运行中、已完成未回流、已回流筛选。真实样本的 106 个分支可直接缩小为 5 个运行中分支和 13 个相关事件。
- Trace 搜索支持用户输入、response、工具、请求编号和派生摘要；异常、慢请求、工具、子 Agent 是直接入口。
- 结果模式直接展示命中的请求，而不是只筛出包含它的整个 Turn；每次生成 24 条结果，并保留 Turn 归属。重复携带的历史 `current_user` 不进入事件搜索索引，避免一条用户问题虚假命中整条长链路。

cursor/turn 分页和增量客户端 store 仍属于后续阶段。

## 1. 审计目标

这次审计不以“界面是否好看”为主要问题，而是检查 peekMyAgent 是否能帮助用户快速回答以下问题：

1. 用户说了什么，Agent 最终做了什么？
2. 模型这一轮真正收到了哪些 system、tools、messages 和 harness 注入？
3. 一次工具调用从发起、执行、结果回传到最终回答是怎样串起来的？
4. Agent 为什么失败、变慢、偏离任务或消耗异常？
5. 多个子 Agent 分别做了什么，哪些完成、哪些回流、哪条链路异常？
6. 哪些内容是原始证据，哪些是重建或启发式推断？
7. 如何把最小、可信的证据分享给另一位开发者？

最终目标不是展示尽可能多的数据，而是让用户先看懂执行故事，再按需下钻到完整证据。

## 2. 方法与样本

本次使用真实浏览器操作了两条现有 Trace：

| 样本 | 规模 | 主要用途 |
| --- | ---: | --- |
| 普通长会话 | 73 requests、9.05 MiB | 普通对话、工具链、上行展开、Raw、轮次导航和三栏布局 |
| 深度研究 Trace | 1,536 requests、108 个子 Agent 实例、9,838 次工具调用、136.05 MiB | 大 Trace 渐进加载、106 子 Agent 并发、最新轮次和极端信息密度 |

实测视口包括 1920×1080 和 1280×720。操作覆盖：

- 会话切换；
- 时间线浏览和轮次导航；
- `tool_use -> tool_result -> final response`；
- 上行构成展开；
- System 翻译、原文和区块搜索；
- Response / Tool use / Tools schema 证据切换；
- 2 个和 106 个子 Agent 的看板与单分支下钻；
- “只看最新轮次”；
- 会话信息和来源说明；
- 左栏、Raw 栏的折叠与宽窄屏布局。

含真实提示词、路径和结果的截图仅保存在本机 `.local/ux-audit-2026-07-11/`，不进入公开仓库。

审计同时检查了当前 Viewer 的加载、渲染和布局代码。Impeccable 附带的确定性 UI detector 在本机报告 `bundled detector not found`，因此本报告不声称包含该 detector 的自动规则结果；视觉结论来自真实浏览器操作、截图和 DOM 量化。

## 3. 总体判断

当前产品已经是一个能力完整的 **Trace 证据查看器**，但还不是一个足够高效的 **Trace 分析工作台**。

它最强的部分是证据完整性：上行构成、原始 JSON、翻译、工具调用和多 Agent 归属都已经能看到。它最弱的部分是分析顺序：页面仍然倾向于把所有可用字段平铺出来，让用户自己完成归因。

一句话概括现状：

> peekMyAgent 已经能回答“数据在哪里”，下一阶段需要优先回答“发生了什么、为什么、哪里值得看”。

## 4. 已经做得好的部分

### 4.1 上行构成已经形成产品壁垒

“System / Tools / History / 本轮新增 / 厂商 token 统计”的结构清楚，能把模型上下文从一个巨大 JSON 变成可理解的组成。固定内容默认折叠、详情按需展开，也符合用户的阅读顺序。

### 4.2 Raw Inspector 的证据边界表达成熟

- `Raw reconstructed` 明确说明原始 body 已不可用、当前内容由 blob 和 request tree 重建。
- Assistant 的 Raw 视图优先展示 Response，并保留 `tool_use` 和注入的 Tools schema。
- Tools schema 明确说明它来自上行 harness 注入，而不是 response 返回。
- 区块级搜索、翻译、缓存状态、复制和重译已经形成完整闭环；搜索与原文/翻译操作随分类标签固定，并根据原文/整理视图分别检索原始 JSON 或结构化翻译块。Tools 批量复制保留工具与参数归属。
- 请求卡和上行视图不再混入 Response 或本次响应 `tool_use`；“完整请求”和“请求 Metadata”同样排除 response、响应状态及其派生统计。请求侧使用单层紧凑标签，完整请求在首位、Metadata 在末位。下行信息统一从 Assistant 回复进入，Assistant 的 Tools schema 继续明确标为“上行参考”。

这部分体现了产品对“原始事实”和“派生解释”的尊重，应继续保持。

### 4.3 普通工具链在时间线上基本可读

请求卡按因果顺序展示：用户问题、Assistant 发起 `tool_use`、下一次请求中的 `tool_result`、模型继续行动或生成最终回答。使用协议原名 `tool_use` / `tool_result` 也降低了术语漂移。

### 4.4 小规模多 Agent 概览已经有正确骨架

在 2 个子 Agent 的场景中，当前看板能显示任务、状态、请求数、工具数、spawn/return 和分支顺序。用户能快速获得“分了几路、各自做什么、是否回流”的概念。

### 4.5 焦点布局是有效的

在 1280×720 下同时收起会话栏和 Raw 面板后，时间线顶栏从约 201-226px 恢复到 76px，中间栏占满 1280px，工具链阅读明显顺畅。这证明三栏结构本身可以保留，但必须变成任务驱动的自适应布局。

## 5. 关键实测数据

| 场景 | 结果 | 结论 |
| --- | ---: | --- |
| 大 Trace 首批渲染 | 约 3.2s 后出现 32 个 request article | 首次可见反馈仍偏慢 |
| 后台完整补齐 | 首批出现后约 4.9s 跳到完整页面 | 不是持续分页，而是延迟全量成本 |
| 首批 DOM | 1,759 节点、321 buttons、32 articles | 可接受 |
| 完整 DOM | 73,243 节点、16,377 buttons、1,644 articles | 明显超出工作台合理范围 |
| “只看最新轮次” | 2.64s；67,320 节点、15,127 buttons、1,519 articles | 对 1,413-request 的单轮几乎没有减负 |
| 106 子 Agent 面板 | 12,224px 高、3,024 buttons、1,406 事件、约 50.8 万文本字符 | 当前组件无法扩展到大 fan-out |
| 1280 三栏全开 | 中间栏约 541px，顶栏约 226px | 标题逐字换行，核心操作受挤压 |
| 1280 仅收左栏 | 中间栏约 706px，顶栏仍约 201px | Raw 对照可用，但长标题仍破坏布局 |
| 1280 时间线专注 | 中间栏 1280px，顶栏 76px | 应成为窄屏默认或显式模式 |

当前渐进加载实现会先取 32 条，然后在 40ms 定时器后请求完整 Trace 并再次 `renderAll()`。它改善了空白等待，但没有改变网络、内存和 DOM 最终仍与整条 Trace 线性增长的事实。

## 6. 优先问题

### P1：页面以“完整数据”为渲染单位，而不是以“当前任务”为单位

用户通常只在检查一个轮次、一个工具失败或一个子 Agent，但页面会把整条 Trace 的卡片、按钮和隐藏内容生成到 DOM。大 Trace 的切换、展开和“只看最新轮次”都会触发明显主线程成本。

**建议：**

- Viewer API 改为 turn/cursor 分页；
- 客户端只保留当前窗口和少量前后预取；
- Agent 分支、内部事件和 Raw JSON 都按展开状态懒创建；
- “只看最新轮次”只获取并渲染该轮概览，内部 1,000+ requests 仍分页；
- 单个交互不再默认调用全页 `renderAll()`。

### P1：多 Agent 看板在大 fan-out 下从“概览”退化为“数据墙”

2 个 Agent 时，卡片地图和事件顺序有帮助；106 个 Agent 时，同一组件同时生成 106 个概览卡、1,406 个事件按钮和 106 个详情卡。用户无法回答最重要的问题：哪些失败、哪些仍运行、哪些没有回流、哪一路最慢或最贵。

另外，当前样本中 101 个 Agent 显示“已完成”、5 个显示“运行中”，但整个任务只有 0 条“回流”。“子 Agent 已完成模型工作”和“结果已交还主 Agent”是两个不同状态，当前视觉语言没有充分解释差异。

**建议：**默认由“状态与异常”组织，而不是由“全部事件”组织，详见第 9 节。

### P1：同一指标存在语义不一致，损害观测工具可信度

大 Trace 顶栏显示“子代理 108”，会话信息弹窗显示“子代理 857”。代码上前者优先使用 `subagent_instance_count`，后者来自带 `x-claude-code-agent-id` 的 request 数量。两个值都可能正确，但标签相同。

**建议：**统一为明确指标：

- 子 Agent 实例：108；
- 子 Agent 请求：857；
- 已完成 / 已回流 / 运行中 / 异常：分别统计。

所有面板必须使用同一个统计 DTO 和术语表。

### P1：缺少面向整条 Trace 的搜索、过滤和异常入口

当前搜索只覆盖右侧当前 Raw 区块。面对 1,536 requests，用户不能直接搜索：

- 某个工具名、错误文本、request/call/agent ID；
- 某种角色或 harness 注入；
- 失败、超时、孤儿 tool result、未回流 Agent；
- cache miss、上下文突增、stop reason 异常；
- 仅看用户轮次、仅看工具交换、仅看子 Agent。

没有这些入口时，Turn rail 只能帮助定位顺序，不能帮助定位问题。

### P1：窄屏默认布局没有体现任务优先级

1280px 是开发者常见的半屏或录屏宽度。三栏全开时，固定 292px 左栏和 380-560px Raw 栏把中间主任务压到 541px。响应式切换只在 1080px 以下发生，太晚。

**建议：**使用容器宽度和任务触发的面板策略：

- 小于 1440px：默认只显示时间线；Raw 作为右侧抽屉；
- 1440-1799px：会话栏或 Raw 二选一；点击 Raw 时自动收起会话栏；
- 1800px 及以上：允许三栏并列；
- 用户手动调整仍被记忆，但不能把主栏压到不可读宽度。

### P2：轮次边界仍会被 Harness 注入污染

深度 Trace 中，Skill 的完整说明会成为 Turn 标题；普通会话首个标题包含自动注入的 `Write the title in...`。这些内容不是用户真正的意图，却进入了最高层导航。

**建议：**建立稳定的语义入口类型：

- User input；
- Slash command；
- Harness injection / Skill loaded；
- Tool result；
- Task notification；
- Subagent result；
- Framework internal。

只有 User input 和 Slash command 能开启主 Turn；其他事件归入当前 Turn 的内部时间线。标题使用清洗后的用户意图，不使用完整注入正文。

### P2：信息层级仍偏“字段平权”

每个请求同时给出 System、Tools、Response、Raw，以及耗时、finish、input、cache、output。它们都可用，但对大多数诊断并不等价。用户最先需要的是异常和因果，不是每轮重复看到全部入口。

**建议：**默认展示“本轮故事”，证据按钮收进统一 Inspector；只有发生变化或异常时，System / Tools / token 指标才升级为显著信息。

### P2：固定 composer 会遮挡时间线

底部独立发送框在浏览时持续占用垂直空间，而它与原终端不共享上下文。对只读分析任务，它是低频功能。

**建议：**默认收为单行 command bar 或可折叠 composer；仅在 live、可发送且用户主动进入发送模式时展开。

## 7. 用户真正关心的优先顺序

从高到低建议为：

1. **结果与异常：**完成了吗，哪里失败，是否还在运行？
2. **因果链：**哪条输入触发了哪次模型回复、工具调用、结果和最终回答？
3. **变化：**这一轮相对上一轮新增或改变了什么？
4. **上下文：**模型当时具体看到了哪些 prompt、tools、history 和注入？
5. **多 Agent 分工：**谁负责什么，进展、成本、结果和回流状态如何？
6. **原始证据：**如果解释有疑问，能否回到完整原文和 JSON？
7. **全局统计：**请求总数、Raw 大小、累计工具数。

当前界面把第 4、6、7 项做得很强；下一阶段应把第 1、2、3、5 项提升到主路径。

## 8. 推荐的信息架构

不建议推翻三栏。推荐把它改造成三层语义，而不是三个并列数据区。

### 第一层：Trace Overview

选择会话后先回答：

- 捕获来源、协议、模型和证据置信度；
- 用户轮次、子 Agent 实例、运行/完成/回流/失败；
- 失败工具、慢请求、上下文突增、cache miss 等异常；
- 最贵或最慢的 Turn / Agent；
- 可搜索、可过滤的全局入口。

这里不展示全部请求卡，只展示可行动的摘要。

### 第二层：Turn Story

中间栏默认按用户意图组织一轮完整故事：

```text
用户意图
  -> 模型决定调用工具 / 启动子 Agent
  -> 工具执行结果 / 子 Agent 进展
  -> 模型继续行动
  -> 最终回答或异常结束
```

内部请求默认折叠为一条因果链。用户切换“详细时间顺序”时，才展开每个 request。

### 第三层：Evidence Inspector

右栏保持现有优势，但按三个语义组整理：

- **Context**：System、Tools schema、Harness、Messages、Diff；
- **Exchange**：Request、Response、`tool_use`、`tool_result`；
- **Evidence**：完整 Raw、Metadata、Provenance。

这样可以减少 9 个同级 tab 带来的选择成本，同时不丢失任何证据。

## 9. 多 Agent 推荐方案

### 9.1 默认视图：按 Agent 汇总

顶部先显示一个非常紧凑的状态条：

```text
106 agents | 101 completed | 5 running | 0 returned | 1,406 requests | 4,294 tool calls
```

紧接着提供：状态、Agent 类型、任务、工具、耗时、是否回流等筛选。

主体建议使用可排序、可虚拟化的列表或表格，而不是 106 张同权卡片：

| Agent | Task | Status | Requests | Tools | Duration | Return |
| --- | --- | --- | ---: | ---: | ---: | --- |
| 子 1 | 研究端粒损耗 | Running | 8 | 6/5 | 42s | -- |
| 子 2 | 搜索经典论文 | Completed | 12 | 9/9 | 18s | Not returned |

默认排序：Failed > Running > Completed but not returned > Returned。

### 9.2 下钻视图：单 Agent 本地链路

展开一行后只渲染这个 Agent：

```text
spawn -> model request -> tool_use -> tool_result -> model reply -> return
```

每一步显示状态、耗时和一行摘要；点击后在右栏打开完整证据。不要同时创建其他 105 个 Agent 的详情 DOM。

### 9.3 辅助视图：按时间顺序

时间交错视图仍然有价值，但应作为切换项：

- “按 Agent”用于理解分工；
- “按时间”用于理解并发、竞争、先后和回流。

按时间视图使用 swimlane：一条主 Agent lane，加当前筛选出来的子 Agent lanes。超过 8 条时不同时绘制全部 lane，先按异常或用户选择缩小范围。

### 9.4 状态语义必须拆开

建议状态机：

```text
spawned -> running -> completed -> returned
                    \-> failed
                    \-> cancelled
                    \-> orphaned / unpaired
```

“completed”只表示子 Agent 自身结束；“returned”表示结果已进入主 Agent 上下文。两者不能共用一个“已完成”视觉状态。

### 9.5 不建议的方案

- 不把全局 Agent graph 作为默认首页；节点多时只会变成线团。
- 不为每个 Agent 分配永久颜色；106 个 Agent 时颜色不再可辨。
- 不在概览层平铺全部 request index 和 event pill。
- 不默认展开所有 Agent 详情。

## 10. 三个候选演进方案

### 方案 A：保持现有三栏，补焦点模式与懒渲染

改动最小，适合第一个 PR：

- 自适应收起侧栏；
- 全局搜索与过滤；
- Agent 面板虚拟化；
- request/Raw 按需渲染；
- 修复统计语义和 Turn 标题清洗。

优点是风险低；缺点是“字段平铺”的心智模型仍存在。

### 方案 B：Overview -> Story -> Evidence

这是推荐的目标形态。它保留现有能力，但重排用户路径：

- Overview 找问题；
- Story 理解因果；
- Evidence 验证事实。

优点是更接近成熟 observability 产品；缺点是需要稳定 Trace domain DTO，不能只改 CSS。

### 方案 C：多 Agent graph-first

可作为未来的专业视图，不适合作为默认主界面。对 2-8 个 Agent 很直观，对 100+ Agent 需要聚类、过滤和虚拟化，否则会再次形成视觉噪声。

**最终建议：**近期用方案 A 解除性能和布局阻塞，同时按方案 B 设计数据契约；方案 C 只作为多 Agent 的可选视图。

## 11. 建议实施顺序

### PR 1：可信指标与语义清理

- 统一 Agent instance/request/returned 统计；
- 清洗自动 title prompt；
- Skill / Harness / task notification 不再开启主 Turn；
- 增加来源、重建和启发式关联的统一标签。

### PR 2：真正的可见窗口数据路径

- turn/cursor API；
- 客户端增量 store；
- 去掉首屏后自动全量请求和全量 `renderAll()`；
- latest-only 只加载当前 Turn 概览。

### PR 3：任务驱动布局

- 1280/1440/1800 的自适应 pane 策略；
- 时间线、证据对照、发送三个显式焦点模式；
- composer 默认紧凑；
- pane 使用容器条件而不是只看 viewport。

### PR 4：多 Agent 可扩展视图

- 状态摘要和筛选；
- 虚拟化 Agent 表；
- 单 Agent 懒展开；
- 按 Agent / 按时间切换；
- completed 与 returned 状态拆分。

### PR 5：Trace 搜索与异常索引

- 文本、tool、request/call/agent ID 搜索；
- status、role、source、injection、stop reason 过滤；
- 工具失败、长耗时、上下文突增、cache miss、未配对事件检测；
- 搜索结果直接跳转到 Turn Story 和 Evidence。

## 12. 验收指标

### 性能

- 1,500-request Trace 首个可操作内容在本机目标环境中小于 800ms；
- 首屏不自动下载或解释完整 Trace；
- 初始 DOM 少于 5,000 节点；
- 选择 Turn、打开 Raw、切换筛选的可感知阻塞小于 100ms；
- 100+ Agent 概览不生成全部事件和详情 DOM。

### 信息理解

新用户应能在 30 秒内完成：

1. 找到本轮最终回答；
2. 找到一次失败工具及其输入/结果；
3. 说清模型这一轮看到的 System / Tools / Messages；
4. 找到一个仍运行或未回流的子 Agent；
5. 判断当前证据是 exact、reconstructed 还是 heuristic。

### 一致性

- Header、会话信息、Agent 面板的同名指标必须相同；
- 中英文术语和状态机一一对应；
- 每个可见派生结论都能跳回原始证据；
- 每个“完成”状态都能说明完成的是哪个阶段。

## 13. 设计健康评分

| Nielsen 启发式 | 评分（0-4） | 主要原因 |
| --- | ---: | --- |
| 系统状态可见性 | 3 | live、来源、翻译缓存和 Agent 状态可见，但大 Trace 加载阶段不够透明 |
| 匹配真实世界 | 3 | 使用 `tool_use` 等原始协议语义，但部分内部事件仍被当成用户 Turn |
| 用户控制与自由 | 3 | 可折叠、可调整 pane、可返回 Raw；缺少显式焦点模式和全局过滤 |
| 一致性与标准 | 2 | 同名“子代理”指标含义不同，状态术语混用 |
| 错误预防 | 2 | 来源边界说明较好，但异常和未配对事件缺少主动提示 |
| 识别优于记忆 | 3 | 上行摘要和工具链可见；大 Agent 列表仍要求用户自己记忆关系 |
| 灵活与高效 | 2 | 小 Trace 高效，大 Trace 搜索、筛选和虚拟化不足 |

## 14. 结论

下一步不应优先继续增加 Agent 适配器，也不应先做一次纯视觉重绘。最重要的是把已有的丰富数据变成一条可诊断的执行故事，并让成本与当前可见范围相关。

推荐的产品主张可以进一步收敛为：

> 先看懂 Agent 做了什么，再一键下钻它当时真正看到和返回的原始证据。

这能同时保留 peekMyAgent 最独特的“完整原生证据”价值，并解决当前大 Trace 和多 Agent 场景中最明显的认知与性能瓶颈。
